# 2026-03-18

## Branch: fix/issues-48-49-45

Fixed three issues in a single branch:

### #49 — capabilities version mismatch
- Root cause: `fpt-domain` and `fpt-core` had hardcoded `version = "0.1.0"` instead of `version = { workspace = true }`.
- `env!("CARGO_PKG_VERSION")` in `app.rs` read the library crate version, not the CLI binary version.
- Fix: Changed both crates to use workspace version AND made `capabilities()` accept an explicit `cli_version` parameter from the CLI binary.

### #48 — config clear panics with Clap type mismatch
- Root cause: `ConfigClearArgs` defined `--site`, `--auth-mode`, etc. as `bool` flags, which clashed with `ConnectionArgs` (flattened globally with `global = true`) that defined the same long names as `Option<String>`.
- Clap can't have two args with the same `--long` name even if one is global and one is in a subcommand.
- Fix: Replaced individual boolean flags with a single `--fields <name,...>` comma-separated list. Updated capability spec examples accordingly.
- New syntax: `fpt config clear --fields site,auth-mode` or `fpt config clear --all`

### #45 — exit codes for logical failures
- Already substantially addressed. Added defense-in-depth `failure_count > 0` check.
- Added 10 unit tests for `logical_failure_exit_code`.

### Tests added
- 10 unit tests for exit code logic
- 5 CLI contract tests: version match, config clear (all, fields, no-args error, invalid field error)
- All 37 tests pass.
